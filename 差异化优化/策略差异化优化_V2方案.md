# NostalgiaForInfinityX7 策略差异化优化 V2 方案

> 目标文件: `NostalgiaForInfinityX7.py`
> 适用对象: 希望降低“策略拥挤”影响，但又不想大幅破坏 X7 原始风控骨架的使用者
> 设计原则: **低侵入、强可控、易归因、可分阶段回测**

---

## 核心思路

V1 版本的差异化思路，主要集中在“修改入场条件阈值”“增加新指标”“调整大量 grinding 参数”。这些方向并非不能做，但存在两个明显问题：

1. 改动集中在 `populate_entry_trend()` 的信号层，容易破坏 X7 原本最有价值的保护链。
2. 多个参数同时变化后，回测结果一旦漂移，很难归因到底是哪一项优化真正有效。

因此，V2 采用一个更保守、也更适合长期维护的思路：

- **优先做执行层差异化**，而不是先动信号层
- **优先做仓位层差异化**，而不是先动 grinding 深层结构
- **优先做标的层差异化**，而不是先大面积禁用 condition

换句话说，V2 不追求“和别人完全不一样”，而追求：

> 即使看到相似的机会，也不一定在同样的时间、同样的价格、同样的仓位、同样的币种上参与。

---

## 推荐落地顺序

1. **方案 A：执行层差异化**
2. **方案 B：仓位层差异化**
3. **方案 C：标的层差异化**
4. **最后才考虑信号层微调**

这 3 个方案可以分别独立回测，也可以按顺序叠加，但不建议一开始就与 V1 中的大规模信号阈值修改混用。

---

## 方案 A：执行层差异化

### 为什么优先做这个

“策略拥挤”最直接的损失，很多时候不是因为大家看到同样的信号，而是因为：

- 大家在同一根 5m K 线上同时挂单
- 都在非常接近的位置成交
- 热门币上瞬时滑点被放大

如果先从执行层解决，就可以在**不破坏原始 entry logic** 的前提下，减少同频成交。

相比直接修改 `Condition #1~#6` 的阈值，这个方向有三个优势：

- 不动 `populate_entry_trend()` 中的大量保护条件
- 回测行为更稳定，变化更容易解释
- 出问题时更容易回滚

### 建议改动

#### A.1 将固定滑点改为“动态滑点上限”

**当前位置**: [`max_slippage = 0.01`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:787)

当前 X7 使用固定滑点上限：

```python
  max_slippage = 0.01
```

这对不同波动水平的币种并不合理：

- 低波动币 1.0% 太宽
- 高频波动币 1.0% 又可能太窄

更好的方式是引入一个轻量动态规则，例如基于 `ATR_ratio`：

```python
slippage_limit = 0.008
if last_candle["ATR_ratio"] > 0.02:
  slippage_limit = 0.010
if last_candle["ATR_ratio"] > 0.04:
  slippage_limit = 0.012
```

然后在 `confirm_trade_entry()` 的滑点校验中，用动态值替代固定 `self.max_slippage`。

**落地位置**: [`confirm_trade_entry()`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:11465)

#### A.2 按模式区分滑点容忍度

不同模式对滑点的敏感度并不一样：

- `rapid` 更像短线追机会，对成交容忍可以略高
- `grind` 是后续还要管理仓位的模式，执行要更保守
- `normal` 可以维持中性

建议在 `confirm_trade_entry()` 中根据 `entry_tag` 做轻量区分：

```python
if all(c in self.long_rapid_mode_tags for c in entry_tag.split()):
  slippage_limit *= 1.15
elif all(c in self.long_grind_mode_tags for c in entry_tag.split()):
  slippage_limit *= 0.85
```

这个改法的优势是：

- 不改入场信号
- 只改变“是否接受这次成交”
- 很适合单独回测

#### A.3 对热门币采用更严格的执行要求

建议把币种分成两类：

- 超热门币：`BTC / ETH / SOL / XRP / DOGE / BNB / LINK` 一类
- 其他币

对于超热门币：

- 更严格滑点限制
- 更严格成交量质量要求
- 必要时要求更小的初始仓位

这个逻辑不一定要写进信号层，完全可以写在 `confirm_trade_entry()` 里。

### 风险等级

**低到中**

这是 V2 中最值得优先尝试的方向，因为它改变的是“如何成交”，而不是“何时认为市场值得参与”。

### 回测建议

- 先只做 A.1
- 再叠加 A.2
- 最后再看 A.3 是否值得加

不要三项一起上。

## 方案 B：仓位层差异化

### 为什么比深改 grinding 更稳

X7 的 grinding / derisk / stop / leverage 是一整套联动系统，参数分布在：

- [`220~370 行附近`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:220)
- [`528~620 行附近`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:528)
- [`644~664 行附近`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:644)

如果一口气同时改：

- grind thresholds
- profit thresholds
- derisk levels
- leverage
- doom stop

那么回测一旦变化，很难知道到底是哪个变化有效。

相比之下，**仓位层差异化**只做一件事：

> 保持原有信号结构和大部分管理逻辑不变，但让不同模式在“进场时押注多少”上与大众错开。

### 建议改动

#### B.1 热门模式降低首仓权重

优先考虑以下位置：

- [`system_v3_rebuy_mode_stake_multiplier`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:644)
- [`rapid_mode_stake_multiplier_spot`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:651)
- [`rapid_mode_stake_multiplier_futures`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:652)

建议思路：

- `rebuy` 首仓更轻，给后续加仓留更多预算
- `rapid` 首仓也更轻，因为它天然更偏激进
- `normal` 不必先动

示例方向：

```python
system_v3_rebuy_mode_stake_multiplier = 0.20
rapid_mode_stake_multiplier_spot = [0.60]
rapid_mode_stake_multiplier_futures = [0.60]
```

这类修改的好处是：

- 风险含义明确
- 与“策略拥挤”逻辑一致
- 不会直接破坏 protection chain

#### B.2 保持 `grind_mode_max_slots = 1`

当前配置是：

[`grind_mode_max_slots = 1`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:664)

V2 的建议是：

- **保持不变**
- 不把它作为首轮扩容项

原因很简单：

- `_handle_grind_mode()` 只看数量，不看已占用资金，见 [`_handle_grind_mode()`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:11551)
- `Condition #120` 也直接依赖 `num_open_long_grind_mode < self.grind_mode_max_slots`，见 [`Condition #120`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:20596)

在这种结构下，先扩 slots，通常比先调仓位更危险。

#### B.3 先不动 derisk / leverage / doom stop

这是 V2 和 V1 的一个重要区别。

V2 不建议把下面这些作为首轮差异化重点：

- `system_v3_derisk_level_*`
- `futures_mode_leverage`
- `stop_threshold_doom_*`
- `system_v3_stop_threshold_*`

不是因为它们不能改，而是因为它们属于**整套仓位管理的风险轮廓参数**。

更稳妥的顺序应该是：

1. 先改 stake multiplier
2. 单独回测
3. 再决定是否需要继续动 derisk / leverage / stop

### 风险等级

**中**

比执行层更敏感，但仍明显低于“同时深改 grinding + derisk + leverage”的整包做法。

### 回测建议

- 第一轮只改 `stake_multiplier`
- 第二轮再决定要不要评估更深层参数

## 方案 C：标的层差异化

### 为什么比信号层更容易长期维护

如果你的目标是减少“和大众做同样的交易”，那其实有一个很直接的手段：

> 不在所有人最拥挤的币种上竞争。

这比大改入场逻辑的优势是：

- 不破坏 X7 原始信号
- 差异化很直观
- 维护成本低

当前 X7 已经提供了现成结构：

- [`grind_mode_coins`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:665)
- [`top_coins_mode_coins`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:725)

V2 建议继续沿这个方向做，但做法比 V1 更强调“动态筛选思维”。

### 建议改动

#### C.1 静态名单先做“去超热门化”

这一步可以延续 V1 的保守版做法：

- 从 `grind_mode_coins` 中移除最拥挤的一线币
- `top_coins_mode_coins` 保留流动性好的中盘币

这一步仍然有价值，因为实现简单、回测也容易解释。

#### C.2 在模式币池之外，再叠一层流动性质量过滤

不要只依赖静态名单。

建议在进入 `grind` 或 `top_coins` 模式前，再增加轻量质量门槛，例如：

- 最近 288 根空蜡烛不能过多
- 最近 20 根平均成交量不能太低
- 近期波动率不能极端异常

当前代码已经有一些现成基础：

- [`num_empty_288`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:3680)
- `confirm_trade_entry()` 中已有模式入口，见 [`confirm_trade_entry()`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:11465)

所以完全可以在模式确认阶段加一层轻量过滤，而不需要把逻辑塞进每个 condition。

#### C.3 长期目标是“动态名单”，而不是手工常量名单

更成熟的版本可以做成：

- 每周或每月重新筛一次候选币池
- 保留“流动性足够 + 波动可做 + 非最拥挤”的中间层币种

如果暂时不想写自动化筛选代码，也可以先把文档和策略结构留好接口：

- 先保留静态名单
- 后续再迁移到动态名单

### 风险等级

**低到中**

这是非常适合长期维护的一类差异化方案，因为它改变的是“参与哪些战场”，而不是“如何理解市场”。

### 回测建议

- 先只做静态名单精简
- 再加流动性质量过滤
- 最后才评估动态名单

## 为什么 V2 不把“信号层大改”放在前面

V2 不是否认信号层优化的价值，而是认为：

### 1. X7 最难复制的价值，本来就在保护链

真正让 X7 和简单均值回归策略拉开差距的，往往不是某一条 entry logic，而是：

- 全局保护条件
- 多时间框架过滤
- 模式与仓位管理联动

所以如果为了“差异化”而大面积改 `populate_entry_trend()`，很容易把最值钱的部分一起改坏。

### 2. 信号层大改的归因成本最高

比如同时去做这些事：

- 下调 `Condition #1~#4` 阈值
- 引入 ATR / VWAP / volume_ratio
- 改 EMA 周期
- 改 RSI 周期

那么一旦收益变好或变坏，你很难知道真正起作用的是哪一个。

### 3. 很多拥挤问题，本质上不是“信号一样”，而是“成交一样”

这正是 V2 为什么把执行层放在第一位。

## V2 与 V1 的关系

如果你已经有一份 V1 版本的差异化文档，V2 不是取代它，而是给出一个更适合真实落地的收敛版。

### V2 保留的方向

- 精简模式币池
- 适度调整部分模式仓位
- 保守地做少量参数错位

### V2 明显降级的方向

- 大面积改 `Condition #1~#6` 的保护阈值
- 在大量 condition 中同时叠新指标
- 同时深改 `grind + derisk + leverage + stop`

### V2 新增强调的方向

- 动态滑点
- 模式化执行
- 以执行层替代部分信号层差异化

---

## 推荐的实际实施顺序

### 第一阶段

只做下面 3 项：

1. 动态滑点上限
2. `rebuy / rapid` 首仓权重下调
3. 精简 `grind_mode_coins / top_coins_mode_coins`

### 第二阶段

如果第一阶段稳定，再做下面 2 项：

1. 按模式细分滑点容忍
2. 在模式入口增加流动性质量过滤

### 第三阶段

只有在前两阶段都通过后，再考虑：

1. 局部信号层微调
2. 更深层 grinding 参数
3. 动态名单

## 回测与验证要求

每一阶段都应单独验证，不要打包上车。

### 最低验证要求


### 回测要求

每次只改一个包，至少比较：

- 胜率
- Profit Factor
- 最大回撤
- 平均持仓时间
- 总交易数
- 热门币与非热门币的成交分布变化

### 实盘前验证

- dry_run 至少 2~4 周
- 重点观察热门币成交价格、滑点、以及 grind 资金占用变化

## 最终结论

如果目标是“做出差异化，又不把 X7 改得面目全非”，那么 V2 的最佳路线不是先动信号层，而是：

1. **先改执行**
2. **再改仓位**
3. **再改标的**
4. **最后才改信号**

这条路线的优点不是“最激进”，而是：

- 更稳
- 更容易解释
- 更容易回测
- 更容易长期维护

对于 X7 这种超大体量、强风控耦合的策略，这通常比“看起来差异化更强”的做法更有实际价值。
