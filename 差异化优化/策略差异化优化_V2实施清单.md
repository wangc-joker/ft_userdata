## 使用说明

这份清单不是一次性“大改策略”，而是按阶段拆开的最小可行实施路径。

建议严格遵守以下原则：

1. **每一轮只做一个包**
2. **每一轮只改一类行为**
3. **每一轮都单独回测**
4. **不要把执行层、仓位层、标的层一起首轮上线**

如果执行顺序正确，那么即使某一轮效果不好，也能清楚知道是哪一类优化造成的。

---

## 推荐实施顺序

### 第一阶段

只做下面 3 项：

1. 动态滑点上限
2. `rebuy / rapid` 首仓权重下调
3. 精简 `grind_mode_coins / top_coins_mode_coins`

### 第二阶段

如果第一阶段通过，再做：

1. 按模式区分滑点容忍
2. 在模式入口增加轻量流动性质量过滤

### 第三阶段

只有前两阶段稳定后，再考虑：

1. 更细的热门币执行限制
2. 动态币池
3. 局部 signal 微调

## 第一阶段实施清单

## 包 A：动态滑点上限

### 目标

将当前的固定滑点容忍：

[`max_slippage = 0.01`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:787)

改成一个**基于波动率的动态执行阈值**。

这个改动只影响“是否接受成交”，不改变原始入场信号。

### 需要改动的位置

#### A.1 参数区新增动态滑点开关与默认值

**建议插入位置**: [`max_slippage = 0.01`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:787) 下方

**建议新增参数**

```python
  dynamic_slippage_enabled = True
  dynamic_slippage_low_vol = 0.008
  dynamic_slippage_mid_vol = 0.010
  dynamic_slippage_high_vol = 0.012
  dynamic_slippage_atr_mid = 0.02
  dynamic_slippage_atr_high = 0.04
```

### 设计说明

- `ATR_ratio <= 0.02` 使用低波动滑点上限
- `0.02 < ATR_ratio <= 0.04` 使用中波动滑点上限
- `ATR_ratio > 0.04` 使用高波动滑点上限

如果后续回测不理想，这些参数也很容易回滚。

#### A.2 在 5m 指标区增加轻量波动率指标

**建议修改位置**: [`base_tf_5m_indicators()`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:3537)

**建议插入位置**: [`df["num_empty_288"] = ...`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:3680) 之后、全局保护之前

**建议新增代码**

```python
    # Dynamic slippage helper
    df["ATR_14"] = pta.atr(df["high"], df["low"], df["close"], length=14)
    df["ATR_ratio"] = df["ATR_14"] / df["close"]
    df["ATR_ratio"] = df["ATR_ratio"].replace([np.inf, -np.inf], np.nan)
```

### 说明

这里故意只加最小必需指标：

- 不引入 VWAP
- 不引入 volume_ratio
- 不把它扩散到 entry condition

只服务于执行层。

#### A.3 在 `confirm_trade_entry()` 中改成动态滑点判断

**修改位置**: [`confirm_trade_entry()`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:11465)

当前逻辑核心片段是：

```python
        if (side == "long" and slippage < self.max_slippage) or (side == "short" and slippage > -self.max_slippage):
          return True
```

**建议改造方式**

先在 `last_candle = df.iloc[-1].squeeze()` 后计算动态阈值：

```python
        slippage_limit = self.max_slippage
        if self.dynamic_slippage_enabled:
          atr_ratio = last_candle.get("ATR_ratio", np.nan)
          if pd.notna(atr_ratio):
            if atr_ratio > self.dynamic_slippage_atr_high:
              slippage_limit = self.dynamic_slippage_high_vol
            elif atr_ratio > self.dynamic_slippage_atr_mid:
              slippage_limit = self.dynamic_slippage_mid_vol
            else:
              slippage_limit = self.dynamic_slippage_low_vol
```

然后把原来的 `self.max_slippage` 替换为 `slippage_limit`。

### 实施注意事项

- 不要第一轮就叠加“按模式区分滑点”
- 不要第一轮同时改 entry signal
- 如果 `ATR_ratio` 为 NaN，回退到 `self.max_slippage`

### 回测目标

关注：

- 成交取消率是否下降
- 热门币的实际入场数是否更合理
- 总交易数是否变化过大
- 平均滑点附近的订单质量是否改善

## 包 B：首仓权重差异化

### 目标

对容易拥挤的模式做“轻首仓”，而不去大改 grinding / derisk / leverage。

### 需要改动的位置

#### B.1 调整 rebuy 模式首仓

**当前位置**: [`system_v3_rebuy_mode_stake_multiplier = 0.25`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:644)

**建议修改**

```python
  system_v3_rebuy_mode_stake_multiplier = 0.20
```

### 理由

- `rebuy` 本来就预期会发生后续仓位调整
- 把首仓略微调轻，比直接改 rebuy threshold 更稳

#### B.2 调整 rapid 模式首仓

**当前位置**

- [`rapid_mode_stake_multiplier_spot = [0.75]`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:651)
- [`rapid_mode_stake_multiplier_futures = [0.75]`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:652)

**建议修改**

```python
  rapid_mode_stake_multiplier_spot = [0.60]
  rapid_mode_stake_multiplier_futures = [0.60]
```

### 理由

- `rapid` 更激进
- 适合做“更轻的第一枪”，而不是大幅调止损

#### B.3 保持 `grind_mode_max_slots = 1`

**当前位置**: [`grind_mode_max_slots = 1`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:664)

**V2 第一阶段建议**

```python
  grind_mode_max_slots = 1
```

即：**不改**

### 理由

- 当前 `_handle_grind_mode()` 只按数量限制，不按资金占用限制，见 [`_handle_grind_mode()`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:11551)
- 先扩槽位的风险大于先轻仓

### 第一阶段明确不动的内容

首仓权重包第一轮不要碰：

- `grind_*_thresholds_*`
- `system_v3_derisk_level_*`
- `futures_mode_leverage`
- `stop_threshold_doom_*`
- `system_v3_stop_threshold_*`

### 回测目标

重点看：

- 首仓暴露是否降低
- 水下资金占用是否缓和
- 利润率是否因首仓过轻而显著受损

## 包 C：静态币池精简

### 目标

先做一版简单、好解释、容易回测的标的层差异化。

不是让策略只做“冷门币”，而是让它少在最拥挤的币上卷。

### 需要改动的位置

#### C.1 精简 `grind_mode_coins`

**当前位置**: [`grind_mode_coins`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:665)

**建议方向**

从原列表中移除最拥挤的一线币，例如：

- `BTC`
- `ETH`
- `BNB`
- `DOGE`
- `XRP`
- `SOL`
- `DOT`
- `LINK`

保留流动性不错、但拥挤度相对低的中盘币。

#### C.2 精简 `top_coins_mode_coins`

**当前位置**: [`top_coins_mode_coins`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:725)

**建议方向**

- 保留中盘优质币
- 适当减少超高关注度币种比例
- 不要求和 `grind_mode_coins` 完全一致

### 设计原则

- 不要一次删得太狠
- 不要把币池压缩到过度集中
- 优先删“极热+大家都做”的币，不优先删“流动性差”的币

### 第一阶段建议写法

直接采用保守版静态名单，不做动态筛选代码。

也就是说：

- 第一轮只改列表常量
- 不新增自动筛币逻辑

### 回测目标

重点比较：

- 热门币成交占比
- 总交易数变化
- 非热门币表现是否足以覆盖减少的热门币机会

## 包 D：按模式区分滑点容忍

### 前提条件

必须在“动态滑点上限”已经独立验证通过后再做。

### 目标

同样是成交确认，不同模式容忍度略有差异：

- `rapid` 稍微放宽
- `grind` 稍微收紧
- `normal` 保持基准

### 修改位置

仍然在 [`confirm_trade_entry()`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:11465)

在动态滑点计算之后，叠加模式修正：

```python
        if entry_tag is not None:
          tags = entry_tag.split()
          if all(c in self.long_rapid_mode_tags for c in tags):
            slippage_limit *= 1.15
          elif all(c in self.long_grind_mode_tags for c in tags):
            slippage_limit *= 0.85
```

### 注意事项

- 第一轮不要把倍率调太大
- 只允许轻微修正，不要放大成“另一套完全不同的成交系统”

## 包 E：模式入口轻量流动性过滤

### 前提条件

必须在静态币池精简已经验证通过后再做。

### 目标

不是给每个 condition 加新过滤，而是在**模式入口**做轻量过滤。

### 最适合修改的位置

#### E.1 `_handle_grind_mode()`

**位置**: [`_handle_grind_mode()`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:11551)

当前这里只做：

- 是否在币池
- 是否超过 slots

建议再加一层轻量检查，例如：

- 最近 `num_empty_288` 不能过高
- `ATR_ratio` 不能过低或过高

实现方式可以是在函数里重新拿一次 analyzed dataframe：

```python
    df, _ = self.dp.get_analyzed_dataframe(pair, self.timeframe)
    if len(df) >= 1:
      last_candle = df.iloc[-1].squeeze()
      if last_candle.get("num_empty_288", 0) > 0:
        return False
      atr_ratio = last_candle.get("ATR_ratio", np.nan)
      if pd.notna(atr_ratio) and (atr_ratio < 0.008 or atr_ratio > 0.06):
        return False
```

#### E.2 `_handle_top_coins_mode()`

**位置**: [`_handle_top_coins_mode()`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:11565)

同样可以加轻量过滤，但建议比 grind 更宽松。

### 注意事项

- 这里只加模式级过滤
- 不要把它塞回 `populate_entry_trend()`
- 不要第二阶段同时引入 VWAP / volume_ratio / Chaikin

## 最终建议

如果你准备把 V2 真正落地成代码，最佳顺序就是：

1. **先改动态滑点**
2. **再改首仓权重**
3. **再改静态币池**
4. **最后才做模式级附加过滤**

这条路线的最大优点不是“最炫”，而是：

- 风险最可控
- 改动最容易解释
- 回测最容易归因
- 最符合 X7 这种复杂策略的维护方式

如果后续你决定进入“第二阶段代码实现”，这份清单已经足够直接对应到 `NostalgiaForInfinityX7.py` 的具体改动位置。
