# NostalgiaForInfinityX7 策略差异化优化 - 最终融合版

> 目标文件: `NostalgiaForInfinityX7.py`
> 目标: 在不破坏 X7 原始保护链和仓位管理骨架的前提下，做出一版**适合真实落地**的差异化优化
> 文档定位: **最终执行版本**

---

## 适用原则

这份文档不是“所有可行思路的合集”，而是已经收敛后的最终推荐版本。

它综合了以下三类来源：

1. V1 方案中可直接落地、且风险相对可控的参数调整
2. V2 方案中更合理的战略排序（执行层 > 仓位层 > 标的层 > 信号层）
3. 多轮评审中确认的风险边界

最终保留的思路只有一个核心原则：

> **优先改变成交方式、仓位暴露和参与标的，不优先改动 X7 的信号骨架。**

因此，这个“最终融合版”只保留 7 个改动项，并明确划分：

- 哪些属于**首轮必做**
- 哪些属于**首轮可选**
- 哪些**本轮不做**

---

## 首轮实施范围

### 首轮必做

1. 动态滑点上限
2. `rapid / rebuy` 首仓下调
3. `grind_mode_coins` 精简
4. `top_coins_mode_coins` 精简
5. `grind` 的部分 `profit_threshold` 轻微抬高

### 首轮可选

6. 按模式区分滑点容忍
7. 在模式入口增加轻量流动性过滤

### 首轮明确不做

- 不批量改 `Condition #1~#6` 的信号阈值
- 不批量削弱 `RSI_3 > 3.0` 保护链
- 不在多个 condition 中铺开 ATR / VWAP / volume_ratio 新过滤
- 不修改 `grind_mode_max_slots = 1`
- 不首轮修改 derisk
- 不首轮修改 leverage
- 不首轮修改 doom stop / system_v3 stop 体系

## 修改项 1：动态滑点上限

### 目的

解决“大家同一时刻成交”的问题，优先从执行层做差异化。

当前 X7 使用固定滑点：

[`max_slippage = 0.01`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:787)

固定滑点的问题是：

- 对低波动币过宽
- 对高波动币又可能过窄

因此，建议改成基于 `ATR_ratio` 的动态滑点。

### 修改位置 1.1：参数区新增动态滑点参数

**位置**: [`max_slippage = 0.01`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:787) 下方

**原始代码**

```python
  max_slippage = 0.01
```

**修改为**

```python
  max_slippage = 0.01

  dynamic_slippage_enabled = True
  dynamic_slippage_low_vol = 0.008
  dynamic_slippage_mid_vol = 0.010
  dynamic_slippage_high_vol = 0.012
  dynamic_slippage_atr_mid = 0.02
  dynamic_slippage_atr_high = 0.04
```

### 修改位置 1.2：在 5m 指标中增加 ATR_ratio

**位置**: [`base_tf_5m_indicators()`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:3537) 中，
[`df["num_empty_288"] = ...`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:3680) 之后、全局保护之前

**新增代码**

```python
    # Dynamic slippage helper
    df["ATR_14"] = pta.atr(df["high"], df["low"], df["close"], length=14)
    df["ATR_ratio"] = df["ATR_14"] / df["close"]
    df["ATR_ratio"] = df["ATR_ratio"].replace([np.inf, -np.inf], np.nan)
```

### 修改位置 1.3：在 `confirm_trade_entry()` 中使用动态滑点

**位置**: [`confirm_trade_entry()`](/D:/work/NostalgiaForInfinity/NostalgiaForInfinityX7.py:11465)

在：

```python
      last_candle = df.iloc[-1].squeeze()
```

之后加入：

```python
      slippage_limit = self.max_slippage
      if self.dynamic_slippage_enabled:
        atr_ratio = last_candle.get("ATR_ratio", np.nan)
        if pd.notna(atr_ratio):
          if atr_ratio > self.dynamic_slippage_atr_high:
            slippage_limit = self.dynamic_slippage_high_vol
          elif atr_ratio > self.dynamic_slippage_atr_mid:
            slippage_limit = self.dynamic_slippage_mid_vol
          else:
            slippage_limit = self.dynamic_slippage_low_vol
```

并将原先的：

```python
        if (side == "long" and slippage < self.max_slippage) or (side == "short" and slippage > -self.max_slippage):
```

替换为：

```python
        if (side == "long" and slippage < slippage_limit) or (side == "short" and slippage > -slippage_limit):
```

## 修改项 6：轻微抬高 Grind 的部分 `profit_threshold`

### 目的

让止盈节奏与大众稍微错开，但不去首轮深改 `sub_thresholds / derisk / leverage`。

这项是从 V1 中保留下来的高性价比部分。

### 为什么保留这项

相比下面这些高风险改动：

- 改加仓深度
- 改 derisk 层级
- 改杠杆
- 改 doom stop

只轻微调整 `profit_threshold` 的风险明显更低。  
它改变的是“何时止盈”，而不是“何时加仓”或“如何承受更深风险”。

## 首轮不做的内容

这些内容本轮全部排除：

### 不做 1：批量改 `Condition #1~#6`

原因：

- 直接进入 X7 核心保护链
- 归因难
- 回滚成本高

### 不做 2：批量下调 `RSI_3 > 3.0`

原因：

- 这是保护链的核心支点之一
- 属于结构性风控变化，不是简单参数错位

### 不做 3：大面积引入 VWAP / volume_ratio / Chaikin 过滤

原因：

- 归因成本高
- 容易在多个 condition 里产生大面积行为漂移

### 不做 4：修改 `grind_mode_max_slots = 1`

原因：

- 当前 grind 限制只看数量，不看资金占用
- 与 grinding 参数调整叠加后风险放大

### 不做 5：修改 derisk / leverage / doom stop

原因：

- 这些都属于仓位管理轮廓参数
- 不适合和首轮差异化一起打包上线

## 最终结论

这份“最终融合版”的目标，不是追求最强的差异化，而是追求：

- **最适合真实落地**
- **最不容易把 X7 改坏**
- **最容易单独回测归因**

因此，真正值得首轮实施的，就是这 6 项：

1. 动态滑点上限
2. 下调 Rebuy 首仓
3. 下调 Rapid 首仓
4. 精简 `grind_mode_coins`
5. 精简 `top_coins_mode_coins`
6. 轻微抬高部分 Grind `profit_threshold`

如果这 6 项都能稳定通过回测和 dry_run，再考虑模式化滑点与模式入口过滤。

这就是我认为当前阶段，对 X7 最稳、也最有实战价值的一版差异化优化方案。
